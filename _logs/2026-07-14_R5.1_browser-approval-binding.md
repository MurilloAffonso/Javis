# R5.1 — Canais externos, começando pelo browser: aprovação amarrada ao conteúdo

**Data:** 2026-07-14
**Fase:** R5.1 (primeiro canal externo depois do executor)

## O que eu esperava fazer, e o que realmente precisava ser feito

Ia "habilitar o browser": instalar o canal, ligar flags, expor rota. Ao ler o
código antes, o canal já existia e já estava bem gateado — token local, escopo
`project_id`, flag `JAVIS_ENABLE_BROWSER`, approval persistido single-use. O
trabalho de R5.1 não era adicionar canal; era fechar um buraco no gate dele.

## O buraco: a aprovação não olhava o que estava aprovando

`/browser/run` chamava `require_persisted_approval(..., metadata={"task": task})`.
O `metadata.task` era **gravado na criação e nunca reconferido no consumo**.
`repo.approvals.valid_for_action` validava status, uso único, expiração, action,
route e project_id — mas **não o payload**.

Consequência concreta: uma aprovação criada e aprovada para
*"ler a previsão do tempo em weather.com"* podia ser consumida por uma chamada
com o mesmo `approval_id` e task *"entrar no Instagram e postar"*. O humano
aprova uma coisa; a máquina executa outra. Para um canal cuja própria docstring
diz "ação em site real = risco → aprovação humana SEMPRE", isso esvaziava a
aprovação.

Comparação que deixou o gap óbvio: no executor supervisionado o approval é
amarrado ao `spec_hash`; mudar um byte da spec mata o approval. O browser não
tinha o equivalente.

## A correção: `payload_hash`, reusando a coluna que já existe

A tabela `approvals` já tem coluna `spec_hash` (o executor usa). Reaproveitei-a
como binding genérico de payload, sem schema novo. Três mudanças, todas
retrocompatíveis (quem não passa hash continua idêntico):

1. `repositories.valid_for_action(..., payload_hash="")` — quando um hash é
   exigido, o approval só vale se `approval.spec_hash == payload_hash`.
2. `repositories.find_pending_action(..., payload_hash="")` — pendências
   separadas por hash: tarefa diferente ganha a própria linha, não reusa a
   antiga.
3. `gate.require_persisted_approval(..., payload_hash="")` — repassa o hash aos
   dois caminhos (validar e criar; na criação grava em `spec_hash`).

Na rota, `_browser_task_hash(task, project_id)` = sha256 de JSON canônico com
`action`, `project_id` e a task com espaços normalizados. Normalizar espaço é
deliberado: reformatar a mesma tarefa não força reaprovação, só mudança real de
conteúdo invalida. O `project_id` entra no hash para a aprovação nunca vazar
entre escopos.

## Testes

`test_browser_approval_binding.py` — 7 testes:

| Teste | O que prova |
|---|---|
| `test_valid_for_action_exige_hash_igual_quando_pedido` | O predicado recusa hash diferente |
| `test_sem_payload_hash_comportamento_antigo_preservado` | Retrocompat: outras rotas intactas |
| `test_aprovar_tarefa_A_nao_autoriza_tarefa_B` | **O ataque: agente NUNCA é chamado; approval não é consumido** |
| `test_aprovacao_vale_para_a_tarefa_exata_que_foi_aprovada` | Caminho legítimo funciona e é single-use |
| `test_espacos_diferentes_nao_invalidam_a_aprovacao` | Normalização de espaço |
| `test_sem_aprovacao_a_rota_pede_aprovacao_e_nao_executa` | `approved=True` sozinho não basta |
| `test_hash_inclui_projeto_nao_vaza_entre_escopos` | Isolamento por projeto |

Um teste **antigo** (`test_broad_exec_approval.py::test_browser_run_requires_
single_use_approval`) quebrou — e quebrou certo: ele codificava o comportamento
inseguro (aprovação genérica autorizava qualquer task). Atualizado para amarrar
a aprovação ao hash da tarefa, com comentário do porquê. Isso não afrouxou o
teste; alinhou-o ao novo contrato.

Suíte completa: **578 passed, 3 skipped, 0 failed** (eram 571; +7).

## O que NÃO foi feito

- **browser-use / operação real** (cliques em sites) segue exigindo
  `OPENROUTER_API_KEY` + modelo com visão, e continua off por padrão. Não liguei
  nada; só fechei o gate.
- **Telegram, WhatsApp, MCP** — mesma auditoria pendente. A hipótese a testar em
  cada um: o gate valida *o que* está sendo autorizado, ou só *que há* uma
  autorização? O padrão `payload_hash` agora existe para reusar.
