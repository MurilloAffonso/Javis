# R5.1 (continuação) — Auditoria de Telegram e MCP

**Data:** 2026-07-14
**Fase:** R5.1 (canais externos)

## Achado: MCP não tinha gate nenhum

Telegram foi auditado e aprovado (daemon, não rota HTTP; whitelist de chat_id +
intent filtering — risco aceitável). MCP foi crítico: **rota HTTP completamente
aberta**. Sem token local, sem project_id, sem approval. Qualquer requisição
`POST /mcp/{server_id}/call` executava tools remotas.

## Correção: gate completo + amarração de payload

Trazer MCP ao mesmo padrão de browser e executor:

1. **Token local** (`x_javes_local_token`) obrigatório
2. **Project_id scope** obrigatório
3. **Approval persistido single-use** obrigatório
4. **Payload amarrado** ao hash: server_id + tool + arguments

Hash canônico = sha256 de JSON com action/project_id/server_id/tool/arguments.
Trocar qualquer um invalida o approval — mesmo padrão do browser.

Novo model: `MCPCallRequestWithAuth(tool, arguments, project_id, approved,
approval_id)`. Removido o model antigo `MCPCallRequest`.
